# План оптимизации opdsdb.py

## Текущие проблемы:
1. Смешение логики работы с БД и бизнес-логики
2. Использование сырых SQL-запросов вместо ORM
3. Отсутствие транзакционной безопасности в некоторых операциях
4. Нет обработки ошибок при работе с базой данных

## Требуемые изменения:

### 1. Разделить ответственности:
- Вынести бизнес-логику в сервисный слой
- Оставить в opdsdb только низкоуровневые операции с БД
- Создать репозитории для работы с сущностями

### 2. Заменить сырые SQL на ORM:
- Использовать Django ORM для всех операций с БД
- Создать кастомные менеджеры для сложных запросов
- Использовать миграции вместо прямых SQL-запросов

### 3. Улучшить транзакционную безопасность:
- Использовать декораторы transaction.atomic для операций записи
- Реализовать retry-логику для конфликтующих операций
- Добавить блокировки для предотвращения race conditions

### 4. Добавить обработку ошибок:
- Обработка исключений базы данных
- Логирование ошибок с контекстом
- Graceful degradation при ошибках БД

### 5. Оптимизировать производительность:
- Использовать bulk-операции для массовых вставок/обновлений
- Оптимизировать запросы с помощью индексов
- Использовать select_for_update для конкурентных операций

### 6. Улучшить функции работы с архивами:
- Вынести логику работы с архивами в отдельный модуль
- Добавить обработку ошибок при чтении архивов
- Оптимизировать проверку целостности архивов

### 7. Добавить тестирование:
- Unit-тесты для всех функций работы с БД
- Интеграционные тесты с реальной базой данных
- Тесты производительности для критичных операций

### 8. Улучшить документацию:
- Документация для всех функций работы с БД
- Описание транзакционных границ
- Примеры использования функций
