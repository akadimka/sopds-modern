# Техническое задание на разработку SimpleOPDS Catalog NG (новое поколение)

## 1. Введение

### 1.1. Назначение документа
Настоящее техническое задание (ТЗ) определяет цели, требования и архитектурные решения для проекта **SimpleOPDS Catalog NG** (далее – SOPDS‑NG) – форка оригинального проекта SimpleOPDS Catalog, разработанного Дмитрием Шелепневым.

### 1.2. Область применения
SOPDS‑NG представляет собой веб‑приложение для организации OPDS‑каталога электронных книг. Оно предназначено для развёртывания на серверах (включая одноплатные компьютеры Raspberry Pi) и предоставляет пользователям возможность просматривать, искать и скачивать книги через стандартные OPDS‑клиенты. Опционально позволяет использовать в качестве клиента мессенджеры (например Telegram) через дополнительные плагины.

### 1.3. Цели и задачи
- **Основная цель:** Создать современную, поддерживаемую и расширяемую версию оригинального SimpleOPDS Catalog.
- **Ключевые задачи:**
  1. Переход на актуальные версии Python (3.11+) и Django (5.2+).
  2. Реорганизация кодовой базы для улучшения структуры и удобства управления.
  3. Обеспечение работы с PostgreSQL 17 в качестве основной СУБД.
  4. Поддержка запуска в контейнерах Docker/Podman.
  5. Адаптация интерфейса для использования на устройствах с малым экраном (мобильные устройства).
  6. Внедрение современных инструментов разработки: uv, mypy, ruff, pytest.

## 2. Текущее состояние проекта

### 2.1. Реализованная функциональность
- **Полноценный OPDS‑каталог** с поддержкой стандарта OPDS 1.2.
- **Сканирование директорий** и автоматическое извлечение метаданных из файлов книг (FB2, EPUB, MOBI, PDF и др.).
- **Хранение метаданных** в базе данных с возможностью ручной корректировки.
- **Поиск по авторам, названиям, сериям, жанрам** с различными алгоритмами (точное совпадение, по началу строки, вхождение подстроки).
- **Группировка книг** по авторам, сериям, жанрам, каталогам.
- **Поддержка многопользовательского режима** с разграничением прав (администратор, пользователь).
- **Книжная полка пользователя** (bookshelf) с отслеживанием времени чтения.
- **Конвертация форматов** на лету (FB2 → EPUB, FB2 → MOBI).
- **Сложная система обработки дубликатов книг** с возможностью скрытия/показа.
- **Многоязычная поддержка** интерфейса через Django translation.
- **Кэширование** результатов поиска для повышения производительности.
- **Веб‑интерфейс** для управления каталогом и поиска книг.
- **Аутентификация** через HTTP Basic Auth для OPDS и формы входа для веб‑интерфейса.

### 2.2. Технологический стек (реализованный)
- **Язык программирования:** Python 3.11+
- **Фреймворк:** Django 5.2+
- **База данных:** PostgreSQL (основная), SQLite (для разработки)
- **HTTP‑сервер:** Gunicorn (продакшен)
- **Фронтенд:** Django Template Engine, HTML5, CSS3, JavaScript
- **OPDS‑генерация:** Кастомная реализация на основе Atom 1.0
- **Инструменты разработки:** Black, mypy, flake8, pytest (частично)

### 2.3. Архитектурные компоненты
1. **Core** (`opds_catalog/`) – ядро, отвечающее за модели данных, бизнес‑логику и OPDS‑генерацию. Полностью реализовано.
2. **Scanner** – модуль сканирования файловой системы и импорта метаданных. Частично реализован.
3. **Converter** – модуль конвертации книг в различные форматы. Реализована конвертация FB2 → EPUB/MOBI.
4. **Web Interface** (`sopds_web_backend/`) – фронтенд для администратора и пользователей. Полностью реализован.
5. **Authentication** – система аутентификации и авторизации. Реализована через HTTP Basic Auth и Django‑сессии.

## 4. Технические требования

### 4.1. Технологический стек (целевой)
- **Язык программирования:** Python 3.12+.
- **Фреймворк:** Django 5.2+.
- **База данных:** PostgreSQL 17 (основная). Допускается поддержка SQLite для разработки и тестирования.
- **HTTP‑сервер:** Gunicorn (продакшен) + Nginx (обратный прокси).
- **Фронтенд:** HTML5, CSS3, JavaScript. Использование Vue.js и tailwind.css для адаптивного дизайна.
- **Контейнеризация:** Docker с multi‑stage сборкой. Поддержка Podman как альтернативы.
- **Очереди задач:** Celery с брокером Redis. Может быть исключено для работы на одноплатных компьютерах.
- **Кэширование:** Redis (также для сессий) или Memcached для систем с ограничением по ресурсам.
- **Инструменты разработки:**
  - **Менеджер пакетов:** uv.
  - **Форматирование кода:** ruff.
  - **Проверка типов:** mypy,ty.
  - **Линтер:** ruff, isort.
  - **Тестирование:** pytest, coverage.

### 4.2. Поддержка Raspberry Pi
- Приложение должно корректно работать на Raspberry Pi 3/4/5 под управлением Raspberry Pi OS (64‑битная версия).
- Веб‑интерфейс должен быть адаптирован для сенсорного ввода (кнопки и элементы навигации достаточного размера).
- Предоставляются Docker‑образы для архитектур `arm64` и `amd64`.
- Для развёртывания на Pi предлагается использовать docker‑compose с предварительно настроенными конфигурациями.

### 4.3. Требования к производительности
- Загрузка списка авторов (до 10 000 записей) – не более 2 секунд.
- Поиск по каталогу (до 100 000 книг) – не более 1 секунды.
- Одновременная поддержка не менее 20 активных OPDS‑клиентов.
- Фоновая обработка (сканирование, конвертация) не должна блокировать основной интерфейс.

## 5. Архитектура

### 5.1. Компоненты системы (целевые)
1. **Core** – ядро, отвечающее за модели данных, базовую бизнес‑логику и OPDS‑генерацию.
2. **Scanner** – модуль сканирования файловой системы и импорта метаданных.
3. **Converter** – модуль конвертации книг в различные форматы (использует Calibre или сторонние утилиты).
4. **API** – REST API на основе Django REST Framework.
5. **Web Interface** – фронтенд для администратора и пользователей.
6. **Task Queue** – асинхронная обработка задач через Celery.
7. **Authentication** – система аутентификации и авторизации (Django‑сессии + JWT для API).

### 5.2. Структура базы данных
- Основные сущности: **Author**, **Book**, **Series**, **Genre**, **Tag**, **User**, **ScanLog**, **DownloadLog**.
- Связи: многие‑ко‑многим между Book и Author, Book и Genre, Book и Tag.
- Индексы на поля, используемые в поиске (название, автор, серия).

### 5.3. Взаимодействие с внешними системами
- **OPDS‑клиенты** – через стандартные эндпоинты `/opds/`.
- **Внешние метаданные** – через плагины к API Goodreads, Google Books.
- **Уведомления** – через SMTP или Telegram Bot API.

1. Все существующие функции оригинального SimpleOPDS Catalog работают без регрессий.
2. Приложение проходит все автоматические тесты (покрытие кода не менее 80%).
3. Docker‑образ успешно запускается на Raspberry Pi 4 (arm64) и на сервере x86_64.
4. Интерфейс администратора полностью функционален в современных браузерах (Chrome, Firefox, Safari) и на сенсорных экранах.
5. OPDS‑каталог корректно отображается в популярных клиентах (Moon+ Reader, FBReader, Stanza).
6. Документация по развёртыванию и API доступна в репозитории.

## 8. Дополнительные соглашения

- Код должен соответствовать **PEP 8** и форматироваться с помощью **ruff**.
- Все публичные методы и классы должны содержать аннотации типов и docstrings в стиле **ReST**.
- Логирование осуществляется через стандартный модуль `logging`, конфигурация логгера задаётся на уровне приложения.
- Обработка исключений – только конкретные типы, без «голых» `except:`.
