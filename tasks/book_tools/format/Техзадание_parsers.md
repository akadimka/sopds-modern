# Техническое задание на модификацию модуля `parsers`

## 1. Цель

Модернизировать модуль `parsers` пакета `book_tools.format` в соответствии с архитектурными стандартами SOPDS‑NG: завершить переход на единый интерфейс парсеров, определить DTO для передачи данных, устранить дублирование, обеспечить расширяемость и тестируемость.

## 2. Общие архитектурные требования

- **Язык:** Python 3.12+.
- **Стиль кода:** PEP 8, форматирование `ruff`, длина строки до 88 символов, отступы 4 пробела.
- **Типизация:** Аннотации типов для всех публичных функций/методов (включая возвращаемые значения). Использовать `mypy --strict`.
- **Логирование:** Иерархические логгеры через `logging.getLogger(__name__)`, без `print()`.
- **Обработка ошибок:** Конкретные исключения (например, `ParserError`, `UnsupportedFormatError`), обязательная привязка `as e`, логирование/проброс.
- **Документация:** Docstrings в стиле ReST на русском для всех публичных API.
- **Тестирование:** `pytest`, параметризация, покрытие не менее 80%.
- **Зависимости:** Управление через `uv`, минимальные зависимости.

## 3. Конкретные изменения

### 3.1. Трансформация модуля в пакет

Модуль `parsers.py` следует преобразовать в пакет `parsers/` со следующей структурой:

- `__init__.py` — реэкспорт основных компонентов (интерфейс, фабрика, DTO).
- `base.py` — абстрактный базовый класс `EbookParser` и вспомогательные типы.
- `dto.py` — датаклассы для передачи данных (`BookMetadata`, `Cover`, `Author` и т.д.).
- `fb2_parser.py` — реализация парсера для FB2.
- `epub_parser.py` — реализация парсера для EPUB.
- `mobi_parser.py` — реализация парсера для MOBI.
- `factory.py` — фабрика `ParserFactory`.
- `cache.py` — реализация кэширования результатов парсинга.

### 3.2. Создание единого интерфейса парсеров

Разработать абстрактный базовый класс `EbookParser` (или протокол), который определяет общий API для всех форматов книг. Интерфейс должен включать следующие методы:

- `parse_metadata(file_path: Path) -> BookMetadata` — извлечение метаданных книги.
- `extract_cover(file_path: Path) -> Cover` — извлечение обложки (изображение в байтах).
- `extract_content(file_path: Path) -> BookContent` — извлечение полного содержимого книги (опционально, может быть реализовано не для всех форматов).

**Требования к интерфейсу:**

- Все классы парсеров должны быть иммутабельными (не хранить состояние файла).
- Методы принимают путь к файлу (`pathlib.Path`) или файлоподобный объект.
- Каждый парсер отвечает только за один формат.
- Обработка ошибок: при невозможности прочитать файл или некорректном формате выбрасывается исключение `ParserError` (или его подкласс).

### 3.3. Определение DTO (Data Transfer Objects)

Создать набор датаклассов для передачи данных между парсерами и потребителями (фабрика, сервисы, представления). Минимальный набор DTO:

- **`BookMetadata`** — содержит поля:
  - `title: str`
  - `authors: list[Author]` (вложенный датакласс `Author`)
  - `series: Series | None`
  - `genres: list[str]`
  - `language: str`
  - `annotation: str | None`
  - `cover_data: bytes | None` (сырые байты обложки)
  - `publication_date: date | None`
  - `file_size: int`
  - `file_format: str` (например, "fb2", "epub")
  - Дополнительные поля по необходимости.

- **`Author`** — датакласс с полями:
  - `first_name: str`
  - `last_name: str`
  - `middle_name: str | None`
  - `nickname: str | None`

- **`Series`** - датакласс с полями:
  - `name: str`
  - `series_no: int`

Все DTO должны быть сериализуемы (например, через `asdict` из `dataclasses`).

### 3.4. Реализация фабрики парсеров

Создать класс `ParserFactory` со статическим методом `get_parser(file_path: Path) -> EbookParser`. Фабрика должна:

- Определять формат файла по расширению (через `mimetype.py` или собственную логику).
- Возвращать экземпляр соответствующего парсера (например, `FB2Parser`, `EPubParser`, `MobiParser`).
- Если формат не поддерживается — выбрасывать исключение `UnsupportedFormatError`.
- Поддерживать регистрацию новых парсеров через механизм плагинов (например, декоратор или словарь).

### 3.5. Кэширование результатов парсинга

Реализовать простой кэш в памяти (LRU-словарь) для повторного использования уже распарсенных метаданных в рамках одного запроса. Ключ кэша — кортеж `(file_path, mtime)`. Кэшировать только `BookMetadata`, не содержимое книги. Использовать `functools.lru_cache` или самописный LRU-словарь. Кэш должен быть опциональным (включается параметром).

### 3.6. Удаление дублирующихся классов

Удалить все классы с суффиксом `_new` (`Mobipocket_new`, `EPub_new`, `Dummy_new`). Их функциональность должна быть перенесена в соответствующие парсеры, реализующие интерфейс `EbookParser`. Старые имена классов (без `_new`) могут быть сохранены как алиасы для обратной совместимости, но должны делегировать вызовы новым парсерам.

### 3.7. Унификация подхода к парсингу FB2

Выбрать один метод парсинга FB2 — либо `lxml`, либо SAX. Рекомендация: использовать `lxml` (быстрее для полного разбора). SAX оставить только для потоковой обработки очень больших файлов (если они встречаются). Удалить дублирующиеся функции парсинга, оставить только один основной парсер `FB2Parser`, реализующий `EbookParser`.

### 3.8. Разделение больших классов на компоненты (EPub, Mobipocket)

Классы `EPub` и `Mobipocket` (если они существуют) разбить на несколько классов с единственной ответственностью:

- `EpubContainer` — работа с контейнером (извлечение файлов из архива).
- `EpubMetadataParser` — парсинг `content.opf`, `toc.ncx`.
- `EpubCoverExtractor` — извлечение обложки.
- `EpubContentExtractor` — извлечение текста (опционально).

Аналогично для `Mobipocket`. Эти компоненты должны использоваться внутри соответствующего парсера (`EPubParser`, `MobiParser`), но не быть публичным API.

## 4. Требования к тестированию

- Для каждого парсера написать модульные тесты с использованием фикстур (реальные файлы книг в форматах FB2, EPUB, MOBI, PDF — хранить в `tests/fixtures/books/`).
- Тесты должны проверять:
  - Корректность извлечения метаданных (все поля DTO).
  - Корректность извлечения обложки.
  - Обработку битых/некорректных файлов (выброс `ParserError`).
  - Работу фабрики (возврат правильного парсера, выброс `UnsupportedFormatError`).
  - Работу кэша (повторный вызов возвращает тот же объект, инвалидация при изменении файла).
- Для фабрики и кэша — тесты с моками парсеров.
- Покрытие кода модуля `parsers` — не менее 80% (измерять через `pytest-cov`).
- Все тесты запускаются `pytest` без внешних зависимостей (кроме библиотек из `pyproject.toml`).

## 5. Критерии приемки

1. Модуль `parsers` (или пакет) содержит абстрактный базовый класс `EbookParser` с методами `parse_metadata`, `parse_cover`, `extract_content`.
2. Определены DTO: `BookMetadata`, `BookContent`, `Author`, `Chapter` (и другие по необходимости).
3. Реализованы парсеры для форматов FB2, EPUB, MOBI (PDF — опционально), каждый наследует `EbookParser`.
4. Реализована фабрика `ParserFactory` с методом `get_parser`.
5. Реализовано кэширование метаданных (LRU).
6. Удалены все классы с суффиксом `_new`. Старые классы (без `_new`) либо удалены, либо делегируют вызовы новым парсерам.
7. Парсинг FB2 унифицирован (один метод, без дублирования).
8. Классы `EPub` и `Mobipocket` разбиты на компоненты (минимум три компонента для EPUB).
9. Полная типизация (mypy — `--strict` для всех модулей пакета).
10. Docstrings в стиле ReST для всех публичных классов и функций.
11. Тестовое покрытие не менее 80%.
12. Модуль не вызывает регрессий в других частях проекта (сканирование, конвертация).

## 6. Ограничения и замечания

- Изменения не должны затрагивать существующие точки входа (например, функции, вызываемые из модуля сканирования или конвертации). При необходимости — адаптировать их к новому интерфейсу, сохранив обратную совместимость (через депрекацию старых функций).
- Для реализации кэширования не использовать внешние библиотеки (Redis, memcached) — только `functools.lru_cache` или самописный LRU-словарь.
- Приоритеты выполнения: начать с создания интерфейса и DTO, затем реализовать парсеры для каждого формата, затем фабрику и кэш, затем удаление дублирующихся классов и рефакторинг.

## 7. Порядок выполнения

1. **Этап 1 (высокий приоритет):**
   - Создание абстрактного класса `EbookParser` и DTO.
   - Реализация парсера для FB2 (унификация).
   - Реализация парсера для EPUB (с разбивкой на компоненты).
   - Реализация парсера для MOBI.

2. **Этап 2 (средний приоритет):**
   - Создание фабрики `ParserFactory`.
   - Реализация кэширования.
   - Удаление классов `_new` и перенос функциональности.

3. **Этап 3 (низкий приоритет):**
   - Добавление поддержки PDF (если требуется).
   - Оптимизация памяти (потоковый парсинг).
   - Написание тестов для всех компонентов.
