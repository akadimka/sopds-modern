# Техническое задание на изменение пакета `book_tools.format`

## 1. Цель

Привести пакет `book_tools.format` в соответствие с архитектурными стандартами SOPDS‑NG: устранить дублирование, улучшить поддерживаемость, добавить полную типизацию, повысить тестовое покрытие и производительность. Пакет должен стать единым интерфейсом для парсинга и конвертации книг различных форматов (FB2, EPUB, MOBI, PDF и др.).

## 2. Общие архитектурные требования

- **Язык:** Python 3.12+.
- **Стиль кода:** PEP 8, форматирование `ruff`, длина строки до 88 символов, отступы 4 пробела.
- **Типизация:** Аннотации типов для всех публичных функций/методов (включая возвращаемые значения). Использовать `mypy --strict`.
- **Логирование:** Иерархические логгеры через `logging.getLogger(__name__)`, без `print()`.
- **Обработка ошибок:** Конкретные исключения (например, `FormatError`, `UnsupportedFormatError`), обязательная привязка `as e`, логирование/проброс.
- **Документация:** Docstrings в стиле ReST на русском для всех публичных API.
- **Тестирование:** `pytest`, параметризация, покрытие модулей пакета не менее 80%.
- **Зависимости:** Управление через `uv`, минимальные зависимости (только необходимые для парсинга, никаких «тяжёлых» библиотек без надобности).

## 3. Конкретные изменения

### 3.1. Удаление дублирующихся классов (приоритет высокий)

Удалить все классы с суффиксом `_new`: `Mobipocket_new`, `EPub_new`, `Dummy_new`. Их функциональность должна быть перенесена в основные классы (`Mobipocket`, `EPub`, `Dummy`) или заменена новым единым интерфейсом. Устранение дублирования кода, упрощение поддержки.

### 3.2. Создание единого интерфейса для парсеров (приоритет средний)

Разработать абстрактный базовый класс (или протокол) `BookParser`, который определяет общий API для всех форматов:

- `parse_metadata(file_path: Path) -> BookMetadata`
- `parse_cover(file_path: Path) -> bytes | None`
- `extract_content(file_path: Path) -> BookContent` (опционально)

Классы `FB2Parser`, `EPubParser`, `MobiParser`, `PDFParser` (и т.д.) наследуют от `BookParser`.  
Фабрика `ParserFactory.get_parser(file_path: Path) -> BookParser` возвращает соответствующий парсер по расширению/содержимому файла.

**Требования к интерфейсам:**

- Все классы парсеров должны быть иммутабельными (не хранить состояние файла).
- Методы принимают путь к файлу (`pathlib.Path`) или файлоподобный объект.
- Возвращаемые объекты метаданных — датаклассы с полями: `title`, `authors`, `series`, `genres`, `language`, `annotation`, `cover_data` и т.д.

### 3.3. Разделение больших классов (EPub) на компоненты (приоритет средний)

Разбить класс `EPub` (и аналогичные) на несколько классов с единственной ответственностью:

- `EpubContainer` — работа с контейнером (извлечение файлов из архива).
- `EpubMetadataParser` — парсинг `content.opf`, `toc.ncx`.
- `EpubCoverExtractor` — извлечение обложки.
- `EpubContentExtractor` — извлечение текста (опционально).

Аналогично для `Mobipocket` (если есть большие классы).

### 3.4. Унификация подхода к парсингу FB2 (приоритет средний)

Выбрать один метод парсинга FB2 — либо `lxml`, либо SAX.  
Рекомендация: использовать `lxml` (быстрее для полного разбора). SAX оставить только для потоковой обработки очень больших файлов (если они встречаются).  
Удалить дублирующиеся функции парсинга, оставить только один основной парсер `FB2Parser`.

### 3.5. Модуль aes.py (приоритет средний)

Модуль `aes.py` был восстановлен и теперь содержит рабочую функцию `encrypt_file`.  
Требуется:

- Убедиться, что он не используется в других частях проекта (импорты отсутствуют). Если не используется — удалить.
- Если используется — перенести в общий модуль криптографии (`src/sopds_crypto/aes.py`) и обновить импорты.
- Заменить зависимость от `pycryptodome` на `cryptography` (соответствует конвенциям проекта).

### 3.6. Исправление mimetype.py (выполнено)

Модуль `mimetype.py` уже доработан:

- Добавлена полная типизация.
- Исправлена логика получения MIME-типа (используется `dict.get` с fallback на `OCTET_STREAM`).
- Добавлена поддержка точек в расширении.
- Добавлены docstrings.

Дальнейших изменений не требуется.

### 3.7. Фабрика книг (приоритет средний)

Создать класс или функцию `BookFactory.create(file_path: Path, metadata_only: bool = True)`.  
Фабрика:

- Определяет формат файла.
- Выбирает соответствующий парсер через `ParserFactory`.
- Парсит метаданные.
- Возвращает экземпляр `Book` (или датакласс `BookInfo`) с заполненными полями.
- Не создаёт экземпляр модели Django — только представление данных.

### 3.8. Оптимизация памяти (приоритет низкий)

- Для больших файлов (PDF, EPUB) использовать потоковый парсинг (SAX для XML, генераторы для текста).
- Не загружать всё содержимое книги в память при извлечении метаданных.
- Оценить необходимость использования `mmap` для чтения больших MOBI/PDF.

### 3.9. Кэширование результатов парсинга (приоритет низкий)

Реализовать простой кэш в памяти (словарь LRU) для повторного использования уже распарсенных метаданных в рамках одного запроса.  
Ключ — путь к файлу + время модификации `(file_path, mtime)`.  
Не кэшировать содержимое книги, только метаданные.

## 4. Требования к тестированию

- Для каждого парсера написать модульные тесты с использованием фикстур (реальные файлы книг в форматах FB2, EPUB, MOBI, PDF — хранить в `tests/fixtures/books/`).
- Тесты должны проверять: корректность извлечения метаданных, обложки, обработку битых файлов, выброс исключений.
- Для фабрики и единого интерфейса — тесты с моками парсеров.
- Покрытие кода пакета — не менее 80% (измерять через `pytest-cov`).
- Все тесты запускаются `pytest` без внешних зависимостей (кроме библиотек из `pyproject.toml`).

## 5. Критерии приемки

1. Удалены все классы с суффиксом `_new`. Старые имена классов (без `_new`) продолжают работать и содержат полную функциональность.
2. Реализован единый интерфейс `BookParser` и фабрика `ParserFactory`. Все существующие форматы (FB2, EPUB, MOBI, PDF) поддерживаются.
3. Класс `EPub` разбит на компоненты (минимум три класса).
4. В пакете нет дублирующихся функций парсинга FB2.
5. Модуль `aes.py` либо удалён, либо перенесён в общий модуль криптографии с заменой библиотеки `pycryptodome` на `cryptography`.
6. `mimetype.py` корректно определяет MIME-типы для всех поддерживаемых форматов (проверено).
7. Полная типизация (mypy — `--strict` для всех модулей пакета).
8. Docstrings в стиле ReST для всех публичных классов и функций.
9. Тестовое покрытие не менее 80%.
10. Пакет не вызывает регрессий в других частях проекта (сканирование, конвертация).

## 6. Ограничения и замечания

- Изменения не должны затрагивать существующие точки входа (например, функции, вызываемые из модуля сканирования или конвертации). При необходимости — адаптировать их к новому интерфейсу, сохранив обратную совместимость (через депрекацию старых функций).
- Для реализации кэширования не использовать внешние библиотеки (Redis, memcached) — только `functools.lru_cache` или самописный LRU-словарь.
- Приоритеты выполнения пунктов указаны в плане оптимизации. Начать с высокоприоритетных (3.1, 3.6), затем средние, затем низкие.

## 7. Порядок выполнения

1. **Этап 1 (высокий приоритет):**
   - Удаление классов `_new`.
   - Перенос функциональности в основные классы.
   - Проверка и доработка `mimetype.py` (уже выполнено, проверить).

2. **Этап 2 (средний приоритет):**
   - Создание единого интерфейса парсеров и фабрики.
   - Разделение больших классов на компоненты.
   - Унификация парсинга FB2.
   - Решение вопроса с `aes.py`.

3. **Этап 3 (низкий приоритет):**
   - Оптимизация памяти.
   - Кэширование результатов парсинга.
   - Добавление тестов для новых компонентов.
